Before understanding the **module pattern**, you need to understand two things:

Lexical Scope → Closure → Module

That’s the path.

---

### First, what a module is NOT

Many developers group functions and data inside an object:

```javascript
const workshop = {
  teacher: "Kyle",
  ask(question) {
    console.log(this.teacher, question)
  }
}
```

This is called the **namespace pattern**.

It organizes code, but it is **not a module**.

Why?

Because **everything is public**.

---

### What makes something a module?

A real module requires **encapsulation**.

Encapsulation simply means:

• Some things are **public**
• Some things are **private**

This idea is also called **information hiding**.

---

### The classic Module Pattern

```javascript
const workshop = (function () {

  let teacher = "Kyle"   // private state

  function ask(question) {
    console.log(teacher, question)
  }

  return {
    ask
  }

})()
```

Now we have a module.

---

### Why this works

The inner function `ask` **closes over** the variable `teacher`.

That means:

`teacher` remains accessible to `ask`,
but **not accessible from the outside**.

```javascript
workshop.ask("What is closure?")
```

Works.

```javascript
workshop.teacher
```

❌ Undefined.

Private state is protected.

---

### Why closure is essential

Modules **cannot exist without closure**.

Closure preserves access to variables inside the module scope.

Without closure, private state would disappear.

---

### Modules manage state

A module isn't just a container of functions.

A module **tracks state over time**.

If there is no changing state, it's probably just an **over-engineered namespace**.

---

### Module Factory Pattern

Modules don’t have to be singletons.

You can create multiple instances:

```javascript
function createWorkshop(teacher) {

  return {
    ask(question) {
      console.log(teacher, question)
    }
  }

}

const js = createWorkshop("Kyle")
const react = createWorkshop("Suzy")
```

Each module has its **own private state**.

---

### Key Principle

Expose **only what is necessary**.

Everything else stays private.

This is called the **Principle of Least Exposure**.

---

### Why modules matter

Modules are one of the **most important code organization patterns** in JavaScript.

They power:

• Encapsulation
• State management
• Cleaner APIs
• Scalable architecture

Even modern ES Modules conceptually rely on the **same closure idea**.
